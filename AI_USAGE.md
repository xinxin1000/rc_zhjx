# AI 使用说明

## AI 提供的帮助

- 搭建代码结构：配置加载、路由、投递服务、模板渲染、记录存储、管理页面。
- 抽象 `RecordStore` 接口，实现内存存储和 MySQL/GORM 两种实现。
- 设计失败处理策略：立即重试、退避重试、死信告警。
- 设计 replay 接口，支持按条件扫描失败记录重放，包含 `dry_run` 预览。

## 未采纳的建议

- 未采纳完整微服务拆分，保持单进程 MVP。
- 未采纳 Kafka、RabbitMQ 等中间件，使用内存队列，同时提供 MySQL 可选。
- 未采纳复杂模板语言，只保留 `{{a.b}}` 路径占位符。
- 未采纳 Saga 框架，replay 是补偿重放，不涉及多步骤事务。

## 关键决策

- 从 PDF 作业描述提炼需求，确认核心是"配置化 HTTP 通知投递服务"。
- 投递语义采用"至少一次"，依赖外部 API 幂等兜底。
- 失败判断同时看 HTTP 状态码和业务 `code` 字段。
- 内存 store 作为默认实现，MySQL 体现生产演进方向。
- replay 默认只重放 `dead` 状态，避免与退避定时器冲突。
- replay 增加 `dry_run`，避免误操作导致大量重放。
